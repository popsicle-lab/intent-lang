# 设计决策记录

> 记录 intent-lang 设计过程中的关键选择及其理由。

---

## 决策 1："Intent" 的理解方式

我们评估了三种方向：

| 方向 | 描述 | 确定性 | 易用性 |
|------|------|--------|--------|
| **A: 纯自然语言** | 用户写自然语言，系统翻译为逻辑 | ⭐ | ⭐⭐⭐⭐⭐ |
| **B: 纯声明式** | 结构化语法 (require/ensure) | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
| **C: 混合** ✅ | 结构化语法 + LLM 辅助生成 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |

**选择 C 的理由**：

- 方向 A 的歧义与"formally provable"矛盾 —— 同一句话 LLM 可能给出不同形式化结果
- 方向 B 学习成本偏高，非形式化方法背景的用户难以入手
- 方向 C 取两者优点：核心是确定性的结构化语法，LLM 只是"草稿生成器"，生成的代码必须通过 SMT 验证

```
确定性 ──────────────────────→
         B 纯声明    C 混合 ✅    A 纯NL
         学习成本高   平衡点       歧义风险大
←──────────────────────── 易用性
```

## 决策 2：验证层次

| 层次 | 代表工具 | 自动化 | 保证强度 | 实现难度 |
|------|---------|--------|---------|---------|
| **L1 类型检查** | TypeScript, Rust | ⭐⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐ |
| **L2 SMT 求解** ✅ | Dafny, intent-lang | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| **L3 模型检查** | TLA+, Alloy | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ |
| **L4 定理证明** | Lean, Coq | ⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |

**选择 L2 的理由**：

- **vs L1**：类型检查无法检测逻辑错误（`transfer(-100)` 类型正确但逻辑错误）
- **vs L3**：模型检查受限于有界状态空间，变量多时状态爆炸
- **vs L4**：需要手写证明，学习曲线陡峭，与"只声明意图"的理念冲突
- **L2 的甜点**：用户不写证明，solver 自动求解；无界验证；失败时给反例

**演进路径**：

```
当前: L1 (类型检查) + L2 (SMT)
  → Phase 2: + L3 可选模式 (intent check --bounded 100)
  → Phase 3: + L4 辅助 (对 unknown 结果提供 proof hint)
```

## 决策 3：实现语言

| 语言 | 性能 | WASM | Z3 集成 | 开发速度 |
|------|------|------|---------|---------|
| **Rust** ✅ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ (`z3` crate) | ⭐⭐⭐ |
| TypeScript | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐ (CLI only) | ⭐⭐⭐⭐⭐ |
| Go | ⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐ (CLI only) | ⭐⭐⭐⭐ |

选 Rust：性能最优、WASM 支持最好（未来 Playground）、Z3 有成熟的 Rust bindings。

## 决策 4：领域扩展方式

| 方式 | 描述 | 灵活性 | 安全性 |
|------|------|--------|--------|
| 内置所有领域类型 | 核心包含所有类型 | ⭐ | ⭐⭐⭐⭐⭐ |
| **插件架构** ✅ | 核心不变，领域通过插件加载 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| DSL 宏 | 用户自定义语法 | ⭐⭐⭐⭐⭐ | ⭐⭐ |

选插件架构：核心语言保持简洁，任何领域通过 `import` 插件获得类型 + 安全规则 + 公理。

> 详见 [插件系统设计](../architecture/PLUGINS.md)

---

## 决策总结

| 维度 | 选择 | 一句话理由 |
|------|------|-----------|
| Intent 表达 | 混合 (C) | 结构化保证安全 + LLM 降低门槛 |
| 验证层次 | SMT (L2) | 自动化高 + 保证强 + 工程可行 |
| 实现语言 | Rust | 性能 + WASM + Z3 bindings |
| 领域扩展 | 插件架构 | 核心不变，领域无限扩展 |
